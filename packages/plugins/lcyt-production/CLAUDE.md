# `packages/plugins/lcyt-production` — Production Control Plugin (v0.1.0)

Express router plugin for camera PTZ presets, video mixer source switching, and hardware encoder control. Used as an internal dependency by `lcyt-backend` (imported as `lcyt-production`).

**Main entry:** `src/api.js`
**Usage in lcyt-backend:**
```js
import { createProductionRouter, initProductionControl, createPerceptionManager, DEFAULT_PREVIEW_BASE_URL } from 'lcyt-production';
const { registry, bridgeManager, mediamtxClient } = await initProductionControl(db);
// BACKEND_URL (this backend's real, externally-reachable URL — same var
// routes/video.js hands the DSK renderer), not DEFAULT_PREVIEW_BASE_URL
// (localhost-only) — a perception job may run on a remote worker.
const perceptionBackendUrl = process.env.BACKEND_URL || DEFAULT_PREVIEW_BASE_URL;
const perceptionManager = createPerceptionManager({ previewBaseUrl: perceptionBackendUrl, callbackBaseUrl: perceptionBackendUrl });
app.use('/production', createProductionRouter(db, registry, bridgeManager, { publicUrl, mediamtxClient, perceptionManager }));
```

**Source files (`src/`):**
- `api.js` — `initProductionControl(db)` → `{ registry, bridgeManager, mediamtxClient }` (the MediaMTX client is non-null only when `MEDIAMTX_API_URL` is set) + `createProductionRouter(db, registry, bridgeManager, opts)`, which mounts `/cameras`, `/mixers`, `/encoders`, and `/bridge` subrouters. Also re-exports `crud.js`'s plain camera/mixer CRUD functions for `packages/lcyt-tools`, and re-exports `OBSClient` for `lcyt-bridge`'s `ObsPool`.
- `registry.js` — `DeviceRegistry`: loads cameras and mixers from DB, manages live adapter connections, resolves adapters by device type. Also owns production-follow subscription (plan_vertical_crop.md §4): `onProgramChanged(cb)`/`onCameraPresetRecalled(cb)` (subscribe, return an unsubscribe fn) + `notifyProgramChanged(data)`/`notifyCameraPresetRecalled(data)` (fire — listener errors are caught/logged per-listener, never thrown). Fired from `routes/mixers.js`'s `POST /:id/switch/:inputNumber` and `routes/cameras.js`'s `POST /:id/preset/:presetId` **after either transport succeeds** (direct via this registry, or bridge-relayed) — a bridge-relayed switch never calls `switchSource()`/`callPreset()` at all, so the route, not those methods, is the one place that sees both paths. Payload's `apiKey` is the acting session's (`req.session.apiKey`), not a device-ownership lookup — `prod_mixers`/`prod_cameras` aren't project-scoped. `notifyCameraPresetRecalled()`'s `preset` field is **not** the raw `:presetId` route param (a UUID/`.id`) — `routes/cameras.js` recomputes the same presetNumber-or-array-index key `lcyt-web`'s crop editor binds `crop_source_map.camera_preset` to, since the two never matched otherwise (fixed 2026-07-20, see CONSIDER.md and `lcyt-rtmp`'s CLAUDE.md). `lcyt-backend/src/server.js` wires both callbacks to `lcyt-rtmp`'s `CropManager.applyForSource()`. **Not wired:** `lcyt-tools`' `mixer.switch`/`camera.preset` tool handlers call `switchSource()`/`callPreset()` directly, bypassing these routes — an AI-tool/MCP-driven switch or preset recall does not fire production-follow (see CONSIDER.md).
- `crud.js` — Plain, directly-callable camera/mixer CRUD (`listCameras`/`createCamera`/`updateCamera`/`deleteCamera` + mixer equivalents) — the in-process counterpart to `routes/cameras.js`/`routes/mixers.js`'s HTTP handlers, for callers with no Express `req`/`res` (`packages/lcyt-tools`'s `camera.*`/`mixer.*` tools, `plan/mcp`). Also owns `buildSwitchCommand(mixer, inputNumber)`, extracted here so both `routes/mixers.js` and the shared tool registry dispatch bridge-relayed mixer switches through one implementation rather than two.
- `perception-manager.js` — `createPerceptionManager({ previewBaseUrl, callbackBaseUrl, env })` / `isPerceptionDispatchAvailable(env)`: fps30 tracker job dispatch (`plan_video_perception.md` Phase 2 Stream B). `start(apiKey, camera, { emitIntervalMs })` POSTs a `{ id, type: 'perception', apiKey, cameraId, feedKind: 'dedicated', frameUrl, callbackUrl, internalToken, emitIntervalMs }` plan to `ORCHESTRATOR_URL`'s `/compute/jobs` (preferred, `X-Internal-Auth`) or, if unset, directly to `WORKER_DAEMON_URL`'s `/jobs` (`X-Worker-Auth`) — the exact same two dispatch knobs `FFMPEG_RUNNER=worker` already uses, deliberately not a third config surface (see `lcyt-worker-daemon`'s `perception/runner.js` module doc for why perception has no 'local'/'docker' mode). `frameUrl` is built from the camera's own `cameraKey` against `previewBaseUrl` — a dedicated-feed camera's `cameraKey` IS its MediaMTX path name, so the already-public `GET /preview/:key/incoming` route works unmodified. Throws `{code: 'NOT_CONFIGURED'}` with neither URL set, `{code: 'NO_FEED'}` for a camera with no `cameraKey`. **Idempotent:** a second `start()` for a key that already has a running job returns `{jobId, alreadyRunning: true}` without dispatching a duplicate — without this, a retried start (client timeout, double form-submit) would leak an orphaned, unstoppable duplicate job. `stop(cameraId)`/`status(cameraId)` track jobs in an in-memory `Map` keyed by camera id; `stop()` only clears the tracked entry once the remote `DELETE` is confirmed (2xx or 404 — job already gone), returning `false` and leaving the entry tracked on any other failure so the caller can see it's still (possibly) running and retry, rather than silently reporting success. **Phase 3 — shared/mixer-only cameras:** `startSharedFeed(apiKey, { emitIntervalMs })`/`stopSharedFeed(apiKey)`/`sharedFeedStatus(apiKey)` dispatch one job per *project* (not per camera) against the project's shared preview feed (`GET /preview/:apiKey/incoming`), with `cameraId: null` and `feedKind: 'shared'` on the plan (a real typed field, not a sentinel `cameraId` string — code-review fix, avoids anything reading `detection.cameraId` downstream mistaking an unresolved detection for a real camera id) — the worker doesn't know which camera that shared feed currently shows; `lcyt-backend`'s `shared-feed-resolver.js` re-tags each detection with the real camera before it reaches the aggregator. Same `running` Map, keyed `shared:${apiKey}` internally so a project's dedicated-camera and shared-feed jobs never collide.
- `bridge-manager.js` — `BridgeManager`: manages SSE connections from `lcyt-bridge` agents. `sendCommand(instanceId, command, { timeoutMs })` — per-call timeout override; defaults to 120s for `type: 'model_call'` (local AI inference is slow, `plan_ai_model_registry.md`) and 10s otherwise. Resolved command results now pass response payload fields (e.g. `status`/`body` from `http_request`/`model_call`) through to the caller instead of dropping them. Sends SSE heartbeats every 20s.
- `db.js` — SQLite migrations for `prod_cameras`, `prod_mixers`, `prod_bridge_instances`, `prod_encoders` tables.
- `camera-thumbnail.js` — `captureCameraThumbnail(db, camera, registry, opts)` + `deleteCameraThumbnailFile(cameraId, thumbnailsDir)` + `thumbnailPath(cameraId, thumbnailsDir)`: still-frame thumbnail capture for a camera's "picture", persisted to disk (`CAMERA_THUMBNAILS_DIR`) + a `thumbnail_captured_at` DB timestamp. Two capture paths depending on the camera's control type — see "Camera Thumbnail Capture" below.
- `obs-client.js` — `OBSClient`: shared OBS WebSocket v5 abstraction (connection lifecycle, auto-reconnect, RPC helpers) used by both the OBS mixer adapter (direct connections) and `lcyt-bridge`'s `ObsPool` (bridged connections). Owns all OBS protocol details.
- `mediamtx-client.js` — `MediaMtxClient` copy for production-control use (path liveness checks, WHIP publisher kick, path pre-registration). Copied from `lcyt-rtmp`'s client to avoid a cross-plugin dependency — keep in sync. Reads `MEDIAMTX_API_URL` / `MEDIAMTX_WEBRTC_BASE_URL` / `MEDIAMTX_API_USER` / `MEDIAMTX_API_PASSWORD`.
- `routes/cameras.js` — CRUD + PTZ preset trigger + thumbnail capture/serve (`POST /:id/thumbnail/capture`, `GET /:id/thumbnail[.jpg]`). `GET /` and `GET /:id` include a computed `thumbnailUrl` (`null` until first capture) and a computed `live` field for any `camera_key`-bearing camera. Accepts an optional `opts.auth` — see "Camera ownership"/"Camera CRUD auth wiring" below. Also `POST /:id/perception/start|stop`, `GET /:id/perception/status` (`plan_video_perception.md` Phase 2) — thin wrappers delegating to an injected `opts.perceptionManager` (503 when omitted, e.g. no `ORCHESTRATOR_URL`/`WORKER_DAEMON_URL` configured), gated by the same `canAccessCamera()` ownership check as the rest of this router; `start`'s `apiKey` comes from `req.session.apiKey` (falling back to the camera's own `owner_api_key` for an unauthenticated/legacy setup, 400 if neither is available).
- `routes/mixers.js` — CRUD + source switching (dispatch logic now delegates to `crud.js`'s `buildSwitchCommand`).
- `routes/encoders.js` — Hardware encoder CRUD + control (`GET/POST /production/encoders`, `GET/PUT/DELETE /production/encoders/:id`, `POST /production/encoders/:id/start|stop|test`). Encoder types: `monarch_hd`, `monarch_hdx`. Connection sources: `backend` (server calls the encoder's HTTP API), `frontend` (browser calls it directly; backend only stores config), `bridge` (relayed via a bridge agent `http_request` command).
- `routes/bridge.js` — Bridge instance CRUD + SSE command stream + status callback.
- `adapters/camera/amx.js` — AMX camera adapter (TCP/IP PTZ control).
- `adapters/camera/visca-ip.js` — VISCA-over-IP camera adapter (PTZ control via VISCA protocol).
- `adapters/camera/browser.js` — Browser-based camera adapter (WebRTC / media device).
- `adapters/camera/none.js` — No-op camera adapter (software-only targets).
- `adapters/mixer/roland.js` — Roland video mixer adapter (TCP).
- `adapters/mixer/amx.js` — AMX mixer adapter (TCP).
- `adapters/mixer/atem.js` — Blackmagic ATEM mixer adapter.
- `adapters/mixer/obs.js` — OBS Studio mixer adapter (obs-websocket).
- `adapters/mixer/lcyt.js` — LCYT software mixer adapter.
- `adapters/mixer/monarch_hdx.js` — Matrox Monarch HDX encoder/mixer adapter.

**Camera control types:** `none`, `amx`, `visca-ip`, `webcam`, `mobile`, `rtmp` (`webcam`/`mobile` route through the browser/WebRTC adapter; `rtmp` — plan_ingest_feeds.md §1a — is a named feed pushed via RTMP instead, no PTZ, no browser adapter. `webcam`/`mobile`/`rtmp` are the three types that ever get a `camera_key`: the resolvable MediaMTX path identity, populated via WHIP for the first two and via `lcyt-rtmp`'s `feed-rtmp` on_publish resolver for the third — see that plugin's CLAUDE.md. `CAMERA_CONTROL_TYPES` is duplicated between `routes/cameras.js` and `crud.js` (and a third copy of the enum lives in `packages/lcyt-tools/src/tools/cameras.js`); keep all in sync, see CONSIDER.md. `camera_key` values are validated against `CAMERA_KEY_RE = /^[A-Za-z0-9_-]+$/` at create/update time in both `routes/cameras.js` and `crud.js` — it becomes a raw MediaMTX path/shell-command fragment in `lcyt-rtmp`'s `runOnPublish` fan-out registration, so an unsafe value is rejected up front rather than aborting relay start later.)

**Camera ownership (`owner_api_key`, plan_ingest_feeds.md cross-tenant review finding):** `prod_cameras` has no other project/tenant column, so a relay slot's `sourceCameraId` (`lcyt-rtmp`'s `db/relay.js`) could otherwise reference *any* project's camera. `prod_cameras.owner_api_key` (nullable `TEXT`) is stamped automatically from `req.session.apiKey` on `POST /production/cameras` (and accepted as an explicit `ownerApiKey` field by `crud.js`'s `createCamera` for in-process callers with real project context); `NULL` means legacy/unowned/open, matching every camera created before this column existed. `registry.js`'s `parseCamera()` strips the raw `owner_api_key` value out of every API response (it's a real `api_keys.key`) and exposes only a boolean `isOwned`. `routes/cameras.js`'s `canAccessCamera(row, req)` gates `GET /:id`/`PUT /:id`/`DELETE /:id`/preset-trigger/thumbnail-capture to the owning project (or anyone, when no `owner_api_key` is set) and `GET /` filters the list the same way; when no auth middleware is configured at all (`req.session` undefined) every camera stays fully open, preserving pre-existing behavior for callers that don't pass `opts.auth`. `lcyt-rtmp`'s `resolveRelaySourceCameraKey(db, cameraId, requestingApiKey)` enforces the same rule before a relay slot may source from a camera.

**Camera CRUD auth wiring:** `createCamerasRouter(db, registry, bridgeManager, opts)` accepts an optional `opts.auth` (real project-access middleware, e.g. `scopedAuth('production')` — same as every other project-scoped router); `server.js` always supplies it. When supplied, it's applied to every route **except** `isUnauthenticatedCameraRoute()` matches (`/whip`, `/whip-url`, `/thumbnail[.jpg]`) — the WHIP routes back `CameraStreamPage.jsx`, a capability-URL kiosk page with no login flow, and the thumbnail routes are consumed via plain `<img src>` tags that can't carry an Authorization header. **Known gap** (logged in CONSIDER.md): those kiosk pages and the WHIP surface remain fully unauthenticated even with `opts.auth` configured — an existing (but currently disconnected) device-role JWT mechanism (`DeviceLoginPage.jsx` → `POST /auth/device-login`) could close this gap but isn't wired into either kiosk page yet.
**Mixer routes auth wiring (plan_vertical_crop.md §4):** `createMixersRouter(db, registry, bridgeManager, opts)` now also accepts an optional `opts.auth`, mirroring `createCamerasRouter` — previously this router received no auth at all, so `req.session.apiKey` was never available on a mixer switch; needed now so `POST /:id/switch/:inputNumber` can report which project's session performed the switch. `server.js` supplies the same `scopedAuth('production')` to both routers. `isUnauthenticatedMixerRoute()` unconditionally carves out `/whip`, `/whip-url`, and `/sources` — `LcytMixerPage.jsx` (the LCYT software-mixer output kiosk page) plain-`fetch()`s all three with no Authorization header. `/switch/:inputNumber` is a conditional carve-out instead: the same kiosk page also POSTs it with no credentials (to cut its own program source), but the real operator workspace hits the identical route *with* an Authorization header and needs `auth()` to run normally so production-follow gets a real `apiKey` — an unconditional bypass would silently break production-follow for every switch, kiosk or not (a real regression caught by review 2026-07-20, see CONSIDER.md). `hasAuthCredentials(req)` (checks `Authorization`/`x-api-key`/`?token=`/auth cookies) decides which case a given `/switch` request is.

**Camera metadata (plan_video_perception.md Phase 1 Stream A):** `prod_cameras` has three new optional columns — `label` (free text e.g. "pulpit"), `zone` (coarse placement tag: front/back/left/right/wide), and `overlap_links` (JSON array `[{ cameraId, presetId?, kind: 'overlaps_with'|'alternate_for' }]` linking cameras/presets that can cover the same subject). These are returned by `GET /production/cameras` and `/production/cameras/:id`, accepted on `POST`/`PUT` routes, and exposed in `parseCamera()` output as camelCase (`overlapLinks` parsed from JSON). Presets within `control_config.presets[]` may also carry optional `label` and `overlapLinks` fields — those are passed through un-stripped on CRUD, no DB schema needed.

**Camera-to-mixer scoping (`mixer_id`, code-review fix on Phase 3):** `prod_cameras.mixer_id` (nullable `TEXT REFERENCES prod_mixers(id) ON DELETE SET NULL`) records which `prod_mixers` row a camera's `mixer_input` number refers to. Added because `mixer_input` alone is only unique within one mixer — a deployment with more than one mixer could otherwise have two different cameras both claim input 3 with no way to tell them apart. `NULL` means unscoped/legacy (every camera created before this column existed); `lcyt-backend`'s `shared-feed-resolver.js` (`plan_video_perception.md` Phase 3) is the first and only consumer today — it prefers an exact `mixer_id` match and falls back to an unscoped legacy camera only when no scoped match exists. Accepted/returned the same way as `label`/`zone`/`overlap_links` (camelCase `mixerId` in `parseCamera()` output, accepted on `POST`/`PUT /production/cameras[/:id]`).

**Mixer types:** `roland`, `amx`, `atem`, `obs`, `lcyt`, `monarch_hdx`
**Encoder types:** `monarch_hd`, `monarch_hdx` (connection sources: `backend`, `frontend`, `bridge`)

## Camera Thumbnail Capture

`POST /production/cameras/:id/thumbnail/capture` grabs a still frame from a camera's live feed and persists it as that camera's "picture" (`thumbnail_captured_at` on `prod_cameras` + a JPEG file under `CAMERA_THUMBNAILS_DIR`), so an operator can see what a camera is framing without a live preview open. `GET /production/cameras/:id/thumbnail[.jpg]` serves the saved file; `DELETE /production/cameras/:id` removes it too. Manually triggered only — no background/periodic capture.

Both paths fetch from the backend's own public preview-JPEG endpoint (`GET /preview/:key/incoming`, `lcyt-rtmp`'s `PreviewManager`) over plain HTTP, the same "deliberately HTTP, not an in-process import" pattern `lcyt-agent`'s `VisionFrameFetcher` already uses against that endpoint — **this requires `RTMP_RELAY_ACTIVE=1`** on this backend (that's what mounts `/preview` at all); without it every capture attempt fails with the same error a genuinely-offline camera would produce.

- **Path A — independent feed** (`webcam`/`mobile`, `camera.cameraKey` set): fetches `/preview/:cameraKey/incoming` directly. 409 if the camera isn't currently publishing.
- **Path B — program-feed capture** (`amx`/`visca-ip`, no `cameraKey`): these control types are pure PTZ protocols with no video path of their own — the only feed ever available for them is the mixer's program output, and only while this camera is the mixer's active source. The request body must supply `apiKey` (the project's RTMP ingest key; this plugin has no project/apiKey concept of its own, so the caller provides it) and, if more than one mixer row exists, `mixerId` to disambiguate. Liveness is checked via `registry.getActiveSource(mixerId) === camera.mixerInput` before saving — 409 if this camera isn't currently cut to program. Not supported for `none`-type cameras (no `mixerInput` relationship implied) unless `mixerInput` is set.

**Environment variables:**
| Variable | Purpose | Default |
|---|---|---|
| `CAMERA_THUMBNAILS_DIR` | Local directory for captured camera thumbnail JPEGs | `/data/camera-thumbnails` |
| `CAMERA_PREVIEW_BASE_URL` | Base URL this plugin fetches `/preview/:key/incoming` from (same underlying backend process in every real deployment — same convention as `lcyt-agent`'s `VISION_PREVIEW_BASE_URL`/`lcyt-dsk`'s `DSK_LOCAL_SERVER`, each plugin owns its own copy rather than sharing one) | `http://localhost:$PORT` |
